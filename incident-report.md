All activity was intentionally generated by the analyst for the purpose of validating SIEM detection capabilities and practicing incident response workflows. No production systems were involved.

---

## 1. Executive Summary

On April 24, 2026, a simulated SSH brute force attack was executed against a monitored Ubuntu 22.04 endpoint (`ubuntu-agent-01`) in a Docker-based home lab environment. The Wazuh SIEM platform successfully detected the attack across multiple rule levels, correlating individual failed authentication attempts (Rule 5710) into a composite brute force alert (Rule 5712, Level 10). The attack was mapped to MITRE ATT&CK Tactic **Credential Access**, Technique **T1110 - Brute Force**.

In addition, the simulation surfaced secondary findings: file integrity monitoring (FIM) events triggered by file creation, modification, and deletion activity, as well as CIS Ubuntu Linux 22.04 LTS Benchmark compliance failures and known CVEs detected on the agent.

All alerts were detected in real time. No remediation was required given the controlled nature of the exercise.

## 4. Attack Description

### 4.1 SSH Brute Force (Primary Scenario)

An SSH brute force attack was simulated using a shell loop executing repeated non-interactive login attempts against a non-existent user account (`baduser`) on `localhost` (port 22) within the container.

**Attack command used:**
```bash
for i in {1..10}; do ssh -o StrictHostKeyChecking=no -o BatchMode=yes baduser@localhost 2>/dev/null; done
```

The `openssh-server` service was running on the target. Authentication failures were written to `/var/log/auth.log`, which was configured as a Wazuh log source via `ossec.conf`.

### 4.2 File Integrity Monitoring (Secondary Scenario)

A monitored directory (`/tmp/testdir`) was configured in `ossec.conf` with `realtime="yes"`. A test file (`file1.txt`) was created, its permissions modified via `chmod 777`, and then deleted. Each action generated a distinct FIM alert.

## 5. Detection

### 5.1 SSH Brute Force Alerts

| Rule ID | Description | Level | MITRE Tactic | MITRE Technique |
|---------|-------------|-------|--------------|-----------------|
| 5710 | sshd: Attempt to login using a non-existent user | 5 | Credential Access, Lateral Movement | T1110.001, T1021.004 |
| 5712 | sshd: Brute force trying to get access to the system. Non existent user. | 10 | Credential Access | T1110 |

Rule 5712 is a composite rule that fires after 8 or more Rule 5710 events from the same source IP within a 120-second window (`frequency=8`, `timeframe=120`, `same_source_ip=true`).

**Key alert fields (Rule 5712):**
- `data.srcuser`: baduser
- `data.srcip`: ::1 (localhost)
- `location`: /var/log/auth.log
- `rule.frequency`: 8
- `rule.groups`: syslog, sshd, authentication_failures

**Compliance mappings triggered:** PCI DSS 11.4, 10.2.4, 10.2.5 | HIPAA 164.312.b | NIST 800-53 SI.4, AU.14, AC.7 | GDPR IV_35.7.d, IV_32.2

### 5.2 File Integrity Monitoring Alerts

| Rule ID | Description | Level | Event |
|---------|-------------|-------|-------|
| 554 | File added to the system | 5 | /tmp/testdir/file1.txt created |
| 550 | Integrity checksum changed | 7 | Permissions changed (rw-r--r-- to rwxrwxrwx) |
| 553 | File deleted | 7 | /tmp/testdir/file1.txt removed |

FIM alerts included SHA256 hash values, inode, file owner, and permission state before and after change. MITRE mapping: T1565.001 (Stored Data Manipulation), Tactic: Impact.

### 5.3 Secondary Findings

**Vulnerability Detection:** Wazuh syscollector identified 31 CVEs on the agent, including 4 High severity findings (CVE-2022-41409, CVE-2017-11164 affecting libpcre packages) and 22 Medium severity findings across systemd, ncurses, dbus, and tar packages.

**CIS Benchmark (SCA):** The agent scored **46%** (84 passed, 98 failed) against the CIS Ubuntu Linux 22.04 LTS Benchmark v2.0.0. Notable failures included missing separate `/tmp` and `/var` partitions, disabled filesystem integrity checking, and multiple unconfigured sshd hardening settings.

## 6. Response Actions

Given the controlled nature of this simulation, no active remediation was required. The following actions were documented as the expected real-world response:

1. **Containment:** Isolate the affected host from the network to prevent further unauthorized access attempts.
2. **Investigation:** Review `/var/log/auth.log` for full scope of authentication attempts; correlate source IP against threat intelligence feeds.
3. **Hardening:** Apply CIS Benchmark remediation items, particularly sshd configuration hardening (disable root login, set `MaxAuthTries`, configure `LoginGraceTime`).
4. **Patching:** Update vulnerable packages identified by Wazuh vulnerability detection (priority: libpcre, systemd, tar).
5. **Monitoring:** Add active response rule to block source IP after brute force threshold is met.

## 9. Lessons Learned

- Wazuh's composite rule engine (frequency-based correlation) effectively aggregated individual authentication failures into a high-severity brute force alert with no custom rule configuration required.
- Realtime FIM (`realtime="yes"`) provided near-instant detection of file system changes with full metadata including hash values and permission diffs.
- The default Ubuntu 22.04 Docker container image has a low CIS benchmark compliance score, highlighting the gap between a default OS configuration and a hardened production baseline.
- Vulnerability detection triggered automatically upon package installation, demonstrating passive asset monitoring without manual scanning.
